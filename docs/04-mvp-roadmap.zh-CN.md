# MVP 路线图

## 目标

项目初始化完成后，下一阶段应把“生产物料管理系统”的业务设想收敛成可开发的 MVP。MVP 不追求完整 ERP、完整 MES 或完整 WMS，而是先完成生产现场物料闭环的最小可用链路。

## 阶段 0：项目初始化

当前阶段目标：

- 明确系统名称和英文简称。
- 明确系统不做什么。
- 沉淀业务范围、术语关系、数据指标和路线图。
- 建立 Git 仓库和基础 README。

完成标准：

- 根目录存在中英文 README。
- `docs/` 目录存在业务文档入口。
- 文档覆盖系统定位、术语关系、业务范围、数据指标和 MVP 路线。
- Git 仓库已初始化。

## 阶段 1：MVP 需求规格

目标是把业务语言变成开发规格。

建议产出：

- 用户角色清单：生产主管、备料人员、生产人员、仓库人员、绩效核算人员、管理人员。
- 业务流程：备料、领料、用料、剩料、废料、退料、日结、月结。
- 核心单据：生产工单、每日备料单、领料单、用料记录、剩料登记单、废料登记单、退料单、日结表、月结表。
- 单据状态：草稿、已确认、已日结、已月结、已作废。
- 权限边界：谁能录入、确认、调整、查询和导出。

完成标准：

- 每个核心单据都有字段草案。
- 每个流程都有开始条件、结束条件和责任角色。
- 日结后如何调整数据有明确规则。

## 阶段 2：数据模型

目标是设计可落库、可追溯、可统计的数据结构。

建议先设计这些实体：

- 物料
- 产品
- BOM / 标准用料
- 生产工单
- 产线
- 班组
- 员工
- 仓库 / 库位
- 备料单
- 领料单
- 用料记录
- 剩料登记
- 废料登记
- 退料记录
- 日结记录
- 月结记录

完成标准：

- 核心实体有主键、业务编号、状态、创建时间和更新时间。
- 用料、剩料、废料记录必须能追溯到日期、工单、物料、产线和班组。
- 月结数据来源于日结汇总，而不是手工重新填写。

## 阶段 3：技术栈选择

技术栈应服务于业务闭环，不应先于业务模型定型。

需要确认：

- 后端：FastAPI、Django、Spring Boot 或其他。
- 前端：Vue、React、Next.js 或其他。
- 数据库：PostgreSQL、MySQL 或其他。
- 权限：简单角色权限还是组织级权限。
- 部署：本地单机、局域网服务器、Docker Compose 或云服务器。
- 集成：是否需要对接 ERP、MES、WMS。

完成标准：

- 明确开发环境和运行方式。
- 明确数据库和迁移方案。
- 明确第一版是否只做单体应用。

## 阶段 4：工程骨架

目标是生成可运行的最小工程，而不是一次性实现全部业务。

建议顺序：

1. 建立后端项目骨架。
2. 建立数据库迁移机制。
3. 建立前端项目骨架。
4. 实现健康检查、基础导航和基础布局。
5. 实现第一个核心闭环：生产工单到每日备料单。

完成标准：

- 本地能启动后端和前端。
- 有最小测试或检查命令。
- README 记录运行方式。
- 第一个业务闭环可以演示。

## 阶段 5：MVP 闭环

目标是完成第一条可用业务链路：

```text
生产工单
  ↓
每日备料单
  ↓
领料
  ↓
用料
  ↓
剩料 / 废料
  ↓
退料
  ↓
日结
  ↓
月结
```

完成标准：

- 可以录入或导入生产工单。
- 可以生成备料清单。
- 可以登记领料、用料、剩料、废料和退料。
- 可以生成日结数据。
- 可以按月汇总基础指标。
- 可以导出绩效计算所需的基础数据。

## 暂缓事项

第一版暂缓：

- 自动化设备控制。
- 复杂绩效规则引擎。
- 完整采购、销售、财务模块。
- 高级排产。
- 仓库波次和路径优化。
- 多工厂、多组织复杂权限。

这些能力可以在 MVP 跑通后再拆成后续阶段。
